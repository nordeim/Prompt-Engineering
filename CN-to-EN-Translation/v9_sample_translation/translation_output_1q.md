## 3. Project Philosophy and Community

### 3.1 Some Humble Thoughts

- Recently, I discovered that Nature Skills, which I designed, was noticed and referenced by Google DeepMind. They drew inspiration from its citation system, script logic, and skill design philosophy to launch Science Skills. Honestly, this is quite gratifying—when top-tier AI institutions abroad begin to draw inspiration from our work, it shows that the original ideas of Chinese developers are being seen by the world. This is not a sense of loss from being copied, but rather the influence naturally growing outward after Chinese strength has taken root in the open-source soil.
- The core focus of our Skill design has never been to demand that everyone master this philosophy. Rather, the philosophy itself is designed to be understood and reused by machines. If you want to create a brand new Skill or adapt it to your own specific domain, you don't need to start from scratch—simply provide the Nature Skills GitHub URL to Codex, and it will automatically learn the design essence and help you create and modify the new Skill. This is the true liberation of ideas: it no longer relies on word-of-mouth, but flows directly through AI into every corner that needs it.
- The true value of Nature Skills perhaps lies not in those specific skill modules, but in the fact that it quietly opens a new door—it makes many people realize for the first time that they can use Codex or agents to control their local computers for research. I have been fortunate to witness and accompany many people through this paradigm shift in research. The moment they exclaim, "So research can be done this way," that cognitive breakthrough and liberation is far more meaningful to me than the Skills themselves. It is not the success of a tool, but the beginning of a new way of thinking spreading among people.
- Today, almost all practical tools can be distilled into standardized workflows, and standardized workflows can be perfectly encapsulated into reusable skills.

## 4. Quick Start

After installation, you can directly hand over papers, paragraphs, reviewer comments, or task descriptions to the Agent. The following prompts can be copied and used directly:

| What you want to do | Say it directly like this |
|---|---|
| Read papers / bilingual Chinese-English | `Turn this PDF into a bilingual Chinese-English Markdown reader with figure-text alignment.` |
| Generate literature presentation PPT | `Turn this paper into a Chinese group meeting presentation PPT, retaining key figures and source citations.` |
| Polish or translate paper paragraphs | `Rewrite this Chinese paragraph into Nature-style English, keeping the academic meaning unchanged.` |
| Write abstract, introduction, or discussion | `Based on these results and figures, help me draft a Nature-style abstract and introduction.` |
| Pre-submission peer review simulation | `Evaluate this manuscript from a Nature reviewer's perspective and provide three reviewer reports.` |
| Respond to reviewer comments | `Based on this revision email, help me write a point-by-point response, a cover letter, and mark the locations in the revised manuscript that need to be highlighted in red.` |
| Search literature, independent citations, and citer profiles | `Compile the citation count, strict independent citation count, and DOI of this article, and check if there are any academicians, Fellows, or leading experts in the field among the citers.` |
| Create scientific figures or paper schematics | `Based on this methods and results section, help me generate a draft of publication-ready scientific figures or paper schematics.` |

If you are unsure which skill to use, simply describe the task. If you already know the skill name, you can explicitly write "use `nature-reader`" or "use `nature-response`" in the prompt.

## 5. Installation

`nature-skills` is a set of reusable skill packages organized around `SKILL.md`. Each top-level skill directory under `skills/` is an installable unit, such as `nature-*`; `nature-shared` is a shared support package read by other skills and is not counted as an independently triggerable skill in the skills index by default.

### 5.1 `npx skills` Installation Method

You need to install [Node.js 18 or higher](https://nodejs.org/) first. There is no need to install the CLI globally; first check the installable skill names in the repository:

```bash
npx skills add Yuan1z0825/nature-skills --list
```

Install all skills globally to Codex. `nature-shared` will be included with the full installation, so skills that depend on shared reference materials will also work correctly:

```bash
npx skills add Yuan1z0825/nature-skills --global --agent codex --skill '*' --yes --copy
```

When installing a single standalone skill for the current project only, omit `--global`. For example:

```bash
npx skills add Yuan1z0825/nature-skills --agent codex --skill nature-figure --yes --copy
```

When installing `nature-reader`, `nature-paper2ppt`, `nature-polishing`, or `nature-writing` individually, select the shared support package at the same time:

```bash
npx skills add Yuan1z0825/nature-skills --global --agent codex \
  --skill nature-reader --skill nature-shared --yes --copy
```

You can also install all skills to all agents supported by the CLI:

```bash
npx skills add Yuan1z0825/nature-skills --all
```

Check the global installation results and update:

```bash
npx skills list --global --agent codex --json
npx skills update --global --yes
```

Update only one skill, or update only the skills in the current project:

```bash
npx skills update nature-reader --global --yes
npx skills update --project --yes
```

The skill selection parameter uses the frontmatter name displayed by `--list`; for example, the directory `nature-proposal-writer` is currently displayed as `researchwrite`. `npx skills` manages skill files; optional runtime dependencies such as Python, R, browsers, or MCP still need to be configured separately according to the instructions below.

### 5.2 Claude Code Installation Method

Claude Code cannot directly use `scripts/update-codex-skills.sh`, because this script is only responsible for syncing to Codex's `~/.codex/skills/`. When used for Claude Code, it is recommended to keep a stable local clone and then use a subagent or slash command to point to the actual `skills/*/SKILL.md`. This will not破坏 the skill directory structure and allows continued reading of `references/`, `static/`, `manifest.yaml`, scripts, assets, and `skills/nature-shared/`.

If you have not installed Claude Code yet:

```bash
npm install -g @anthropic-ai/claude-code
claude
```

First, clone the repository to a stable path:

```bash
mkdir -p ~/ai-skills
cd ~/ai-skills
git clone https://github.com/Yuan1z0825/nature-skills.git
```

Recommended method: Create a Claude Code subagent wrapper for frequently used skills. Taking `nature-reader` as an example:

```bash
mkdir -p ~/.claude/agents
cat > ~/.claude/agents/nature-reader.md <<'EOF'
---
name: nature-reader
description: Use for Chinese-English paper reading, figure-aware translation, and source-grounded paper notes.
---

When invoked, first read `~/ai-skills/nature-skills/skills/nature-reader/SKILL.md` and follow it as the governing workflow.
Read supporting files from `~/ai-skills/nature-skills/skills/nature-reader/` and `~/ai-skills/nature-skills/skills/nature-shared/` only when needed.
Do not replace this skill with a generic paper-reading response.
EOF
```

Then start a new Claude Code session and directly request the use of this subagent:

```text
Use the nature-reader subagent to turn this paper into a Chinese-English Markdown reader.
```

If you prefer a slash command, you can also create a command wrapper:

```bash
mkdir -p ~/.claude/commands
cat > ~/.claude/commands/nature-reader.md <<'EOF'
Read `~/ai-skills/nature-skills/skills/nature-reader/SKILL.md` first and follow it strictly.
Read directly needed supporting files from `~/ai-skills/nature-skills/skills/nature-reader/` and `~/ai-skills/nature-skills/skills/nature-shared/`.

$ARGUMENTS
EOF
```

Use in Claude Code:

```text
/nature-reader Turn this paper into a complete bilingual Chinese-English Markdown reader.
```

When installing other skills, simply replace `nature-reader` in the example with the corresponding directory name, such as `nature-polishing`, `nature-writing`, `nature-reviewer`, `nature-response`, or `nature-figure`. For subsequent updates, you only need to run:

```bash
cd ~/ai-skills/nature-skills
git pull
```

As long as the wrapper still points to this stable clone path, there is no need to repeatedly copy skill files.

**Automatic Updates (Optional)**

If you want Claude Code to automatically pull upstream updates every time a session starts, you can use `scripts/autoupdate-skills.sh` in conjunction with a `SessionStart` hook.

This method **directly copies** the skills into `~/.claude/skills/` (Claude Code will automatically discover this directory, and skills are loaded directly by directory name), rather than using the wrapper above. Choose one of the two methods.

First, keep a **dedicated** stable clone (used only for syncing skills, please do not make development commits in it):

```bash
mkdir -p ~/ai-skills
git clone https://github.com/Yuan1z0825/nature-skills.git ~/ai-skills/nature-skills
```

For the initial installation, copy the skills into the Claude Code skills directory:

```bash
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --force
```

Then add a `SessionStart` hook in `~/.claude/settings.json` (if `hooks` already exists, merge this item in, do not overwrite the entire file):

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$HOME/ai-skills/nature-skills/scripts/autoupdate-skills.sh",
            "async": true,
            "timeout": 120
          }
        ]
      }
    ]
  }
}
```

`async: true` allows it to run in the background without blocking startup. The script has built-in protection: it defaults to not checking the network repeatedly within 6 hours, automatically skips if the network is down or pulling fails (`exit 0`, never blocking the session), only resyncs when the upstream HEAD actually changes, and refuses to force a fast-forward on a clone with uncommitted changes. The pulled new version will take effect in the **next** session started (the current session's skills are already loaded). The run log is located at `~/.local/state/nature-skills/autoupdate.log`.

The target directory and check frequency are both configurable:

```bash
# Defaults to syncing to ~/.claude/skills; use --dest to point elsewhere, e.g., Codex:
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --dest ~/.codex/skills
# Check at most once per hour:
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --throttle 3600
```

### 5.3 Codex Installation Method

It is recommended to use the repository's built-in script to install or update Codex skills. The script will sync all top-level skill directories under `skills/` and perform a `diff` verification after copying; it will not overwrite other unrelated Codex skills.

```bash
git clone https://github.com/Yuan1z0825/nature-skills.git
cd nature-skills
scripts/update-codex-skills.sh --pull
```

If you have already cloned the repository:

```bash
cd nature-skills
scripts/update-codex-skills.sh --pull
```

Verify whether the current Codex installation is consistent with this checkout:

```bash
scripts/update-codex-skills.sh --check
```

If you use this script for long-term updates and wish to clean up old skill directories that have been deleted upstream:

```bash
scripts/update-codex-skills.sh --pull --prune
```

`--prune` will only delete directories that were previously recorded by this script but are no longer included in the current repository. When run for the first time without history, it will not guess to delete old directories.

You can also hand the repository link to Codex and let Codex execute the installation script. Recommended prompt:

```text
Please install Codex skills from this repository:
https://github.com/Yuan1z0825/nature-skills.git

Please clone the repository and run scripts/update-codex-skills.sh --pull.
After installation, run scripts/update-codex-skills.sh --check to verify.
Please keep the complete skill directories under skills/, do not just copy SKILL.md.
```

If installing only a single skill, please specify the skill name clearly:

```text
Install only nature-reader from this repository:
https://github.com/Yuan1z0825/nature-skills.git

If this skill requires shared files, please also install skills/nature-shared.
```

Key rule: Preserve the complete directory structure. Please copy or reference the entire skill folder, not just `SKILL.md`, because many skills depend on `references/`, `static/`, `manifest.yaml`, scripts, assets, or shared files.

The installation script does not automatically install Python dependencies. When you need to use related scripts or MCP services, install them as needed:

```bash
python -m pip install -r skills/nature-paper-to-patent/requirements.txt
python -m pip install -r skills/nature-paper-to-patent/scripts/disclosure/requirements-cnipa.txt  # Optional: CNIPA publication announcement search
python -m pip install -r skills/nature-academic-search/mcp-server/requirements.txt
```

If you enable the CNIPA publication announcement search for `nature-paper-to-patent`, you also need to execute `python -m playwright install chromium`.

The MCP service for `nature-academic-search` also requires separate configuration of `PUBMED_EMAIL`. Optional providers like Scopus / ScienceDirect require local credential configuration; do not write API keys into repository files.

After installation, please start a new Codex session and then describe the task naturally, for example:

```text
Turn this paper into a complete bilingual Chinese-English Markdown reader.
```

```text
Turn this paper into a Chinese PPT.
```

If you use open-source agent / programming frameworks such as OpenClaw, OpenCode, or Hermes, please see the [OpenClaw / OpenCode / Hermes Integration Tutorial](docs/open-source-agent-frameworks.md).

**Automatic Updates (Optional)**

Codex supports a global `SessionStart` hook. After keeping a dedicated clone, you can check for updates every time you start or resume a Codex session and sync the new version to `~/.codex/skills/`.

First, create a dedicated clone and complete the initial sync:

```bash
mkdir -p ~/.codex
git clone https://github.com/Yuan1z0825/nature-skills.git ~/.codex/.nature-skills-src
~/.codex/.nature-skills-src/scripts/autoupdate-skills.sh \
  --dest ~/.codex/skills --force
```

Then create or merge `~/.codex/hooks.json`:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "/bin/bash \"$HOME/.codex/.nature-skills-src/scripts/autoupdate-skills.sh\" --dest \"$HOME/.codex/skills\"",
            "timeout": 75,
            "statusMessage": "Checking Nature Skills updates"
          }
        ]
      }
    ]
  }
}
```

If there are other hooks in `hooks.json`, please merge the `SessionStart` item, do not overwrite the entire file. After enabling or modifying the hook for the first time, run `/hooks` in Codex to check and trust it. Codex currently executes command hooks synchronously, so here we rely on the script's built-in 6-hour throttling, 60-second network protection, and automatic skip on network failure to avoid repeated network calls or blocked startup due to update failures every session.

The update log is located at `~/.local/state/nature-skills/autoupdate.log`. Pulled new skills usually take full effect in the next session.

## 7. Contribution and Development

### 7.1 Shared Design Principles

All skills adhere to the following principles:

1. **Prioritize primary sources**: Rules are based on published Nature content, official journal guidelines, or explicit local sources, rather than vague aesthetic preferences.
2. **Explicit over implicit**: Every rule should explain the rationale, rather than just giving an assertion.
3. **Awareness of section and task context**: Academic writing, figures, citations, and responses all depend on context; different parts of a paper use different logic.
4. **Output-first**: Every skill should return a ready-to-use product, such as pasteable text, `.svg`, `.pptx`, `.docx`, or specific recommendations.
5. **Extensible**: Each skill is self-contained in its own directory; adding new skills should not require modifying existing skills.

### 7.3 New Skill Addition Process

When adding a skill to this repository, please follow this process:

**1. Create the skill directory**

```text
skills/nature-<topic>/
```

**2. Add required files**

| File | Required | Purpose |
|------|----------|------|
| `SKILL.md` | Required | Frontmatter (`name`, `description`) + rules + workflow; loaded by the agent upon triggering |
| `README.md` | Required | Human-readable documentation (Chinese) |
| `README_EN.md` | Required | Human-readable documentation (English) matching the Chinese page |
| `references/*.md` | Recommended for complex skills | Modular rule files, e.g., APIs, design theories, tutorials, chart types, etc. |

**3. Write bilingual READMEs**

Every new skill must provide both `README.md` and `README_EN.md`. The README is the human-facing skill entry page, not a duplicate of `SKILL.md` or an installation manual. Its goal is to let the user judge within 30 seconds: whether this skill can solve their problem, what they need to provide, what it will produce, and where its boundaries lie.

Basic rules:

- The Chinese README and English README must be one-to-one mirrors: identical number of headings, identical order, identical information points; the English page should not be written as a separate independent template.
- The top must be fixed with the skill name, language switch link, and a one-sentence positioning statement.
- Use the basic structure below by default; insert optional sections only when truly necessary.
- Do not repeat the repository installation tutorial, author information, changelog, development process, complete file tree, or large blocks of internal implementation details in a single skill README.
- Complex rules, API parameters, long tutorials, script instructions, and template indices should be placed in `references/`, `static/`, `scripts/`, or `SKILL.md`; the README should only keep signposts.
- If the skill has visual assets, a small preview table can be included; do not turn the README into a large gallery or a lengthy technical manual.

Chinese README basic structure:

```markdown
# `nature-<topic>` 技能

[English](README_EN.md)

一句话说明这个技能的定位、主要任务和使用边界。

## 适合用它做什么
## 典型请求
## 你需要提供
## 产出
## 边界
## 相关技能
```

The English README must correspond to:

```markdown
# `nature-<topic>` Skill

[中文说明](README.md)

One sentence describing the skill's role, main task, and usage boundary.

## What To Use It For
## Typical Requests
## What You Need To Provide
## Outputs
## Boundaries
## Related Skills
```

Optional sections must be inserted synchronously in both Chinese and English, maintaining the same order. Common optional sections include:

| Chinese Section | English Section | Usage Scenario |
|----------|----------|----------|
| `## 工作方式` | `## Workflow` | Need to explain the core process or routing method |
| `## 运行和依赖` | `## Runtime and Dependencies` | Has scripts, MCP, API keys, local configuration, or external dependencies |
| `## 示例预览` | `## Example Preview` | Has a small number of figures, screenshots, or visual assets worth showing |
| `## 内置参考` | `## Built-In References` | Need to point to `references/`, `assets/`, or demo |
| `## 方法来源` | `## Method Sources` | Writing, review, or analysis rules come from specific sources |
| `## 三种模式` | `## Three Modes` | The skill has clear compose/revise/hybrid modes, etc. |
| `## 与 ... 的关系` | `## Relationship With ...` | Easily confused with another skill, needs to explain the division of labor |

Perform at least these README checks before submitting:

```bash
python scripts/validate-readmes.py
git diff --check
for d in skills/nature-*; do
  [ -f "$d/README.md" ] && [ -f "$d/README_EN.md" ] || continue
  rg -q '^\[English\]\(README_EN\.md\)$' "$d/README.md"
  rg -q '^\[中文说明\]\(README\.md\)$' "$d/README_EN.md"
  test "$(rg -c '^## ' "$d/README.md")" = "$(rg -c '^## ' "$d/README_EN.md")"
done
```
