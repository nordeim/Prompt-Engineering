## 3. Project Philosophy and Community

### 3.1 Some of My Personal Observations

- I recently discovered that the Nature Skills I designed have caught the attention of Google DeepMind, who drew inspiration from its citation framework, scripting approach, and skills design philosophy to launch Science Skills. Honestly, this is quite gratifying — when top international AI institutions begin drawing inspiration from our work, it shows that Chinese developers' original thinking is being recognized by the world. This isn't the disappointment of being copied; it's the natural outward growth of Chinese innovation after taking root in the open-source soil.
- The focus of our Skills design has never been on requiring everyone to fully internalize this philosophy. Rather, the philosophy itself possesses the ability to be understood and reused by machines. If you want to create an entirely new Skill or adapt it to your own specialized domain, you don't need to start from scratch — simply send the Nature Skills GitHub repository to Codex, and it will automatically learn the design principles and help you complete the creation and modification of new Skills. This is the true liberation of ideas: they no longer depend on word-of-mouth transmission but flow directly through AI into every corner that needs them.
- The true value of Nature Skills perhaps lies not in the specific skill modules themselves, but in how it quietly opens a new door — it makes many people realize for the first time that they can use Codex or intelligent agents to operate their local computers for scientific research. I've been fortunate to witness and accompany many people through this paradigm shift in research methodology. The moment they exclaim, "So research can be done this way!" — that cognitive breakthrough and sense of liberation means far more to me than the Skills themselves. It's not the success of a tool; it's a new way of thinking that begins to spread among people.
- In the present day, nearly every practical tool can be distilled into standardized workflows, and standardized workflows can be packaged into reusable skills.

## 4. Quick Start

After installation, you can directly hand papers, paragraphs, reviewer comments, or task descriptions to the Agent. The prompts below can be copied and used directly:

| What You Want to Do | Just Say This |
|---|---|
| Read a paper / bilingual Chinese-English | `Turn this PDF into a bilingual Chinese-English Markdown reader with figure-text alignment.` |
| Generate a literature presentation PPT | `Turn this paper into a Chinese journal club presentation PPT, retaining key figures and source annotations.` |
| Polish or translate a paper paragraph | `Rewrite this Chinese paragraph in Nature-style English while preserving its academic meaning.` |
| Write an abstract, introduction, or discussion | `Based on these results and figures, help me draft a Nature-style abstract and introduction.` |
| Pre-submission review simulation | `Evaluate this manuscript from a Nature reviewer's perspective and provide three reviewer reports.` |
| Respond to reviewer comments | `Based on this revision request email, help me write a point-by-point response, a cover letter, and indicate where the revised manuscript should be marked in red.` |
| Search literature, citations, and citation profiles | `Compile the citation count, rigorous citation count, and DOI of this article, and check whether any Fellows, academicians, or leading figures are among the citers.` |
| Create research figures or paper diagrams | `Based on this methods and results section, help me generate publication-ready research figures or a draft schematic diagram.` |

If you're unsure which skill to use, simply describe your task. If you already know the skill name, you can explicitly state "use `nature-reader`" or "use `nature-response`" in your prompt.

## 5. Installation

`nature-skills` is a collection of reusable skill packs organized around `SKILL.md`. Each top-level skill directory under `skills/` is an installable unit, for example, the `nature-*` skills; `nature-shared` is a shared support package for other skills to read from and is not included in the skills index by default.

### 5.1 `npx skills` Installation

You need [Node.js 18 or higher](https://nodejs.org/) installed first. No global CLI installation is required; first list the available skills in the repository:

```bash
npx skills add Yuan1z0825/nature-skills --list
```

Install all skills globally into Codex. `nature-shared` will be included with the full installation, so skills that depend on shared reference materials will work correctly:

```bash
npx skills add Yuan1z0825/nature-skills --global --agent codex --skill '*' --yes --copy
```

To install a single skill for the current project only, omit `--global`. For example:

```bash
npx skills add Yuan1z0825/nature-skills --agent codex --skill nature-figure --yes --copy
```

When installing `nature-reader`, `nature-paper2ppt`, `nature-polishing`, or `nature-writing` individually, include the shared support package as well:

```bash
npx skills add Yuan1z0825/nature-skills --global --agent codex \
  --skill nature-reader --skill nature-shared --yes --copy
```

You can also install all skills to every agent supported by the CLI:

```bash
npx skills add Yuan1z0825/nature-skills --all
```

Check global installation results and update:

```bash
npx skills list --global --agent codex --json
npx skills update --global --yes
```

Update a single skill, or only skills in the current project:

```bash
npx skills update nature-reader --global --yes
npx skills update --project --yes
```

The skill selection parameter uses the frontmatter name shown by `--list`; for example, the directory `nature-proposal-writer` currently appears as `researchwrite`. `npx skills` manages skill files; optional runtime dependencies such as Python, R, browser, or MCP still need to be configured separately as described below.

### 5.2 Claude Code Installation

Claude Code cannot directly use `scripts/update-codex-skills.sh`, as that script is only responsible for syncing to Codex's `~/.codex/skills/`. For use with Claude Code, it is recommended to keep a stable local clone and then use a subagent or slash command pointing to the actual `skills/*/SKILL.md`. This approach preserves the skill directory structure and continues to access `references/`, `static/`, `manifest.yaml`, scripts, assets, and `skills/nature-shared/`.

If you haven't installed Claude Code yet:

```bash
npm install -g @anthropic-ai/claude-code
claude
```

First clone the repository to a stable path:

```bash
mkdir -p ~/ai-skills
cd ~/ai-skills
git clone https://github.com/Yuan1z0825/nature-skills.git
```

Recommended approach: create a Claude Code subagent wrapper for frequently used skills. Using `nature-reader` as an example:

```bash
mkdir -p ~/.claude/agents
cat > ~/.claude/agents/nature-reader.md <<'EOF'
---
name: nature-reader
description: Use for Chinese-English paper reading, figure-aware translation, and source-grounded paper notes.
---

When invoked, first read `~/ai-skills/nature-skills/skills/nature-reader/SKILL.md` and follow it as the governing workflow.
Read supporting files from `~/ai-skills/nature-skills/skills/nature-reader/` and `~/ai-skills/nature-skills/skills/nature-shared/` only when needed.
Do not replace this skill with a generic paper-reading response.
EOF
```

Then start a new Claude Code session and directly request to use this subagent:

```text
Use the nature-reader subagent to turn this paper into a Chinese-English Markdown reader.
```

If you prefer slash commands, you can also create a command wrapper:

```bash
mkdir -p ~/.claude/commands
cat > ~/.claude/commands/nature-reader.md <<'EOF'
Read `~/ai-skills/nature-skills/skills/nature-reader/SKILL.md` first and follow it strictly.
Read directly needed supporting files from `~/ai-skills/nature-skills/skills/nature-reader/` and `~/ai-skills/nature-skills/skills/nature-shared/`.

$ARGUMENTS
EOF
```

Use it in Claude Code:

```text
/nature-reader Turn this paper into a bilingual Chinese-English Markdown reader.
```

When installing other skills, replace `nature-reader` in the examples with the corresponding directory name, such as `nature-polishing`, `nature-writing`, `nature-reviewer`, `nature-response`, or `nature-figure`. Subsequent updates only require:

```bash
cd ~/ai-skills/nature-skills
git pull
```

As long as the wrapper still points to this stable clone path, there is no need to repeatedly copy skill files.

**Auto-update (optional)**

If you want Claude Code to automatically pull upstream updates each time a session starts, you can use `scripts/autoupdate-skills.sh` with a `SessionStart` hook.

This approach **copies** skills directly into `~/.claude/skills/` (Claude Code automatically discovers this directory and loads skills by directory name), rather than using the wrapper method above. Choose one approach or the other — not both.

First keep a **dedicated** stable clone (used only for syncing skills; do not make development commits in it):

```bash
mkdir -p ~/ai-skills
git clone https://github.com/Yuan1z0825/nature-skills.git ~/ai-skills/nature-skills
```

For the initial installation, copy skills into Claude Code's skills directory:

```bash
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --force
```

Then add a `SessionStart` hook in `~/.claude/settings.json` (if `hooks` already exists, merge this entry rather than overwriting the entire block):

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$HOME/ai-skills/nature-skills/scripts/autoupdate-skills.sh",
            "async": true,
            "timeout": 120
          }
        ]
      }
    ]
  }
}
```

`async: true` lets it run in the background without blocking startup. The script has built-in safeguards: by default it skips network checks within 6 hours of the last check, automatically skips on network failure or pull failure (`exit 0`, never blocks the session), only re-syncs when the upstream HEAD has actually changed, and refuses to force-advance on a clone with uncommitted changes. Newly pulled versions take effect at the **next** session start (the current session's skills have already been loaded). Logs are at `~/.local/state/nature-skills/autoupdate.log`.

Both the target directory and check frequency are configurable:

```bash
# Default syncs to ~/.claude/skills; use --dest to target elsewhere, e.g. Codex:
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --dest ~/.codex/skills
# Check at most once per hour:
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --throttle 3600
```

### 5.3 Codex Installation

The repository includes a built-in script for installing or updating Codex skills. The script syncs all top-level skill directories under `skills/` and performs a `diff` verification after copying; it does not overwrite other unrelated Codex skills.

```bash
git clone https://github.com/Yuan1z0825/nature-skills.git
cd nature-skills
scripts/update-codex-skills.sh --pull
```

If you have already cloned the repository:

```bash
cd nature-skills
scripts/update-codex-skills.sh --pull
```

Verify that the current Codex installation matches this checkout:

```bash
scripts/update-codex-skills.sh --check
```

If you have been using this script for updates long-term and want to clean up old skill directories that have been removed upstream:

```bash
scripts/update-codex-skills.sh --pull --prune
```

`--prune` only deletes directories previously recorded by this script that the current repository no longer contains. On the first run with no history, it will not guess which old directories to delete.

You can also hand the repository link to Codex and let it run the installation script. Recommended prompt:

```text
Please install Codex skills from this repository:
https://github.com/Yuan1z0825/nature-skills.git

Clone the repository and run scripts/update-codex-skills.sh --pull.
After installation, run scripts/update-codex-skills.sh --check to verify.
Please preserve the complete skill directories under skills/; do not copy only SKILL.md.
```

To install only a single skill, specify the skill name explicitly:

```text
Only install nature-reader from this repository:
https://github.com/Yuan1z0825/nature-skills.git

If the skill requires shared files, please also install skills/nature-shared.
```

Key rule: preserve the complete directory structure. Copy or reference the entire skill folder, not just `SKILL.md`, because many skills depend on `references/`, `static/`, `manifest.yaml`, scripts, assets, or shared files.

The installation script does not automatically install Python dependencies. Install them as needed when using the related scripts or MCP services:

```bash
python -m pip install -r skills/nature-paper-to-patent/requirements.txt
python -m pip install -r skills/nature-paper-to-patent/scripts/disclosure/requirements-cnipa.txt  # Optional: CNIPA gazette search
python -m pip install -r skills/nature-academic-search/mcp-server/requirements.txt
```

If enabling the CNIPA gazette search for `nature-paper-to-patent`, you also need to run `python -m playwright install chromium`.

The `nature-academic-search` MCP service additionally requires `PUBMED_EMAIL` to be configured separately. Optional providers such as Scopus / ScienceDirect require local credentials; do not write API keys into repository files.

After installation, start a new Codex session and describe your task naturally, for example:

```text
Turn this paper into a bilingual Chinese-English Markdown reader.
```

```text
Turn this paper into a Chinese PPT.
```

If you use OpenClaw, OpenCode, Hermes, or other open-source agent / programming frameworks, see the [OpenClaw / OpenCode / Hermes Integration Guide](docs/open-source-agent-frameworks.md).

**Auto-update (optional)**

Codex supports a global `SessionStart` hook. After keeping a dedicated clone, you can check for updates at each Codex session start or resume and sync new versions to `~/.codex/skills/`.

First create a dedicated clone and complete the initial sync:

```bash
mkdir -p ~/.codex
git clone https://github.com/Yuan1z0825/nature-skills.git ~/.codex/.nature-skills-src
~/.codex/.nature-skills-src/scripts/autoupdate-skills.sh \
  --dest ~/.codex/skills --force
```

Then create or merge `~/.codex/hooks.json`:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "/bin/bash \"$HOME/.codex/.nature-skills-src/scripts/autoupdate-skills.sh\" --dest \"$HOME/.codex/skills\"",
            "timeout": 75,
            "statusMessage": "Checking Nature Skills updates"
          }
        ]
      }
    ]
  }
}
```

If `hooks.json` already contains other hooks, merge the `SessionStart` entry rather than overwriting the entire file. After enabling or modifying the hook for the first time, run `/hooks` in Codex to check and trust it. Codex currently executes command hooks synchronously, so the script's built-in 6-hour throttle, 60-second network protection, and automatic skip on network failure prevent repeated network checks or startup blocking on update failure at every session.

Update logs are at `~/.local/state/nature-skills/autoupdate.log`. Newly pulled skills typically take full effect in the next session.

## 7. Contributing and Development

### 7.1 Shared Design Principles

All skills adhere to the following principles:

1. **Prioritize primary sources**: Rules are based on published Nature content, official journal guidelines, or explicit local sources, rather than general aesthetic preferences.
2. **Explicit over implicit**: Every rule should state its rationale, not just make assertions.
3. **Perceive section and task context**: Academic writing, figures, citations, and responses all depend on context; different paper sections use different logic.
4. **Output first**: Every skill should return immediately usable deliverables, such as paste-ready text, `.svg`, `.pptx`, `.docx`, or concrete recommendations.
5. **Extensible**: Each skill is self-contained in its own directory; adding new skills should not require modifying existing ones.

### 7.2 Repository Directory Structure

```text
skills/
├── nature-shared/              # Must be preserved when skills reference ../nature-shared
├── nature-<topic>/
│   ├── README.md
│   ├── README_EN.md
│   ├── SKILL.md
│   ├── manifest.yaml     # Included by router-style skills
│   ├── static/           # Included by router-style skills
│   └── references/...
└── nature-proposal-writer/
    ├── README.md
    ├── README_EN.md
    ├── SKILL.md
    ├── scripts/...
    ├── templates/...
    └── references/...
```

### 7.3 New Skill Workflow

When adding a skill to this repository, follow the process below:

**1. Create the skill directory**

```text
skills/nature-<topic>/
```

**2. Add required files**

| File | Required | Purpose |
|------|----------|---------|
| `SKILL.md` | Required | Frontmatter (`name`, `description`) + rules + workflow; loaded by the agent when triggered |
| `README.md` | Required | Human-readable Chinese documentation |
| `README_EN.md` | Required | English documentation matching the Chinese details page |
| `references/*.md` | Recommended for complex skills | Modular rule files, e.g., API, design theory, tutorials, chart types |

**3. Write bilingual READMEs**

Every new skill must provide both `README.md` and `README_EN.md`. The README is a human-facing skill landing page, not a duplicate of `SKILL.md` and not an installation manual. Its goal is to let users determine within 30 seconds: whether this skill can solve their problem, what to provide, what it produces, and where its boundaries lie.

Basic rules:

- The Chinese README and English README must mirror each other one-to-one: same number of headings, same order, same information points; the English page should not be written as a separate independent template.
- The top is fixed as the skill name, a language switch link, and a one-line positioning statement.
- Use the basic structure below by default; insert optional sections only when genuinely necessary.
- Do not repeat repository installation tutorials, author information, changelogs, development history, complete file trees, or large blocks of internal implementation details in an individual skill README.
- Complex rules, API parameters, long tutorials, script documentation, and template indexes should go into `references/`, `static/`, `scripts/`, or `SKILL.md`; the README retains only signposts.
- If a skill has visual assets, a small preview table is acceptable; do not turn the README into a large gallery or lengthy technical manual.

Chinese README basic structure:

```markdown
# `nature-<topic>` Skill

[English](README_EN.md)

One sentence describing the skill's positioning, main task, and usage boundary.

## What To Use It For
## Typical Requests
## What You Need To Provide
## Outputs
## Boundaries
## Related Skills
```

The English README must correspond as:

```markdown
# `nature-<topic>` Skill

[中文说明](README.md)

一句话说明这个技能的定位、主要任务和使用边界。

## 适合用它做什么
## 典型请求
## 你需要提供
## 产出
## 边界
## 相关技能
```

Optional sections must be inserted in sync between Chinese and English and maintain the same order. Common optional sections include:

| Chinese Section | English Section | Use Case |
|----------|----------|----------|
| `## 工作方式` | `## Workflow` | Core process or routing logic needs explanation |
| `## 运行和依赖` | `## Runtime and Dependencies` | Has scripts, MCP, API keys, local configuration, or external dependencies |
| `## 示例预览` | `## Example Preview` | A small number of figures, screenshots, or visual assets worth showing |
| `## 内置参考` | `## Built-In References` | Needs to point to `references/`, `assets/`, or demos |
| `## 方法来源` | `## Method Sources` | Writing, review, or analysis rules come from specific sources |
| `## 三种模式` | `## Three Modes` | The skill has clear compose/revise/hybrid modes |
| `## 与 ... 的关系` | `## Relationship With ...` | Easily confused with another skill; needs differentiation |

Before submitting, at minimum perform these README checks:

```bash
python scripts/validate-readmes.py
git diff --check
for d in skills/nature-*; do
  [ -f "$d/README.md" ] && [ -f "$d/README_EN.md" ] || continue
  grep -q '^\[English\](README_EN\.md)$' "$d/README.md"
  grep -q '^\[中文说明\](README\.md)$' "$d/README_EN.md"
  test "$(grep -c '^## ' "$d/README.md")" = "$(grep -c '^## ' "$d/README_EN.md")"
done
```
