## 3. Project Philosophy & Community

### 3.1 A Few Humble Thoughts

- I recently discovered that Nature Skills, which I designed, has been noticed and referenced by Google DeepMind. They drew on its citation system, scripting ideas, and skill design philosophy to launch Science Skills. Honestly, this is quite gratifying—when a top overseas AI institution starts drawing inspiration from our work, it shows that the original ideas of Chinese developers are being seen by the world. This isn’t the disappointment of being copied; it’s the influence that naturally grows outward after Chinese innovation takes root in open-source soil.
- The focus of our Skills design has never been to require everyone to thoroughly digest this philosophy. Rather, the philosophy itself has the ability to be understood and reused by machines. If you want to create a brand-new Skill or adapt it to your own specialized domain, you don’t need to learn everything from scratch—just send the Nature Skills GitHub URL to Codex, and it can automatically learn the design essence within and help you create and modify your new Skill. This is the true liberation of ideas: no longer reliant on word-of-mouth, it flows directly through AI into every corner that needs it.
- The real value of Nature Skills may not lie in those specific skill modules, but in quietly opening a new door—it allowed many people to realize for the first time that they could use Codex or agents to control their local computers for research. I am fortunate to have witnessed and accompanied many through a paradigm shift in their research. The moment they exclaimed, "So research can be done this way," that cognitive breakthrough and liberation feels far more meaningful to me than the Skills themselves. It isn’t the success of a tool, but a new way of thinking beginning to spread among people.
- In the present day, practically every useful tool can be distilled into a standardized process, and standardized processes can conveniently be encapsulated into reusable skills.

## 4. Quick Start

Once installed, you can hand a paper, a paragraph, reviewer comments, or a task description directly to the Agent. The following prompts can be copied and used directly:

| What You Want to Do | Just Say This |
|---|---|
| Read a paper / side-by-side Chinese-English | `Turn this PDF into a figure-corresponding, side-by-side Chinese-English Markdown reader.` |
| Generate a literature report PPT | `Turn this paper into a Chinese group meeting presentation PPT, retaining key figures and source labels.` |
| Polish or translate a passage | `Rewrite this paragraph in Nature-style English, preserving the academic meaning.` |
| Write an abstract, introduction, or discussion | `Based on these results and figures, help me draft a Nature-style abstract and introduction.` |
| Pre-submission review simulation | `Evaluate this manuscript from a Nature reviewer's perspective and provide three reviewer reports.` |
| Respond to reviewer comments | `Based on this revision email, help me write a point-by-point reply, cover letter, and indicate where the revised manuscript needs to be highlighted in red.` |
| Search literature, citations, and citer profiles | `Compile the citation count, strict self-citation count, and DOI for this article, and check if any of the citers are academicians, Fellows, or big names in the field.` |
| Create scientific figures or conceptual diagrams | `Based on this method and results, help me generate a publication-quality scientific figure or conceptual diagram draft.` |

If you’re unsure which skill to use, simply describe the task; if you already know the skill name, you can explicitly write “use `nature-reader`” or “use `nature-response`” in the prompt.

## 5. Installation

`nature-skills` is a set of reusable skill packs organized around `SKILL.md`. Each top-level skill directory under `skills/`, such as `nature-*`, is an installable unit; `nature-shared` is a shared support pack read by other skills and, by default, is not counted as an independently triggerable skill in the skill index.

### 5.1 Installing via `npx skills`

Requires [Node.js 18 or later](https://nodejs.org/). No global CLI installation is needed. First, view the installable skill names in the repository:

```bash
npx skills add Yuan1z0825/nature-skills --list
```

Install all skills globally for Codex. `nature-shared` will be included with the full installation, so skills that depend on shared reference materials will also work properly:

```bash
npx skills add Yuan1z0825/nature-skills --global --agent codex --skill '*' --yes --copy
```

To install only a single skill for the current project, omit `--global`. For example:

```bash
npx skills add Yuan1z0825/nature-skills --agent codex --skill nature-figure --yes --copy
```

When installing `nature-reader`, `nature-paper2ppt`, `nature-polishing`, or `nature-writing` alone, also select the shared support pack:

```bash
npx skills add Yuan1z0825/nature-skills --global --agent codex \
  --skill nature-reader --skill nature-shared --yes --copy
```

You can also install all skills for all agents supported by the CLI:

```bash
npx skills add Yuan1z0825/nature-skills --all
```

Check global installation results and update:

```bash
npx skills list --global --agent codex --json
npx skills update --global --yes
```

Update only one skill, or only skills in the current project:

```bash
npx skills update nature-reader --global --yes
npx skills update --project --yes
```

The skill selection parameter uses the frontmatter name displayed by `--list`; for example, the directory `nature-proposal-writer` currently appears as `researchwrite`. `npx skills` manages the skill files. Optional runtime dependencies like Python, R, browsers, or MCP still need to be configured separately as described below.

### 5.2 Installing for Claude Code

Claude Code cannot directly use `scripts/update-codex-skills.sh`, as that script only syncs to Codex’s `~/.codex/skills/`. For Claude Code, it’s recommended to keep a stable local clone and then use a subagent or slash command that points to the actual `skills/*/SKILL.md` files. This preserves the skill directory structure and allows continued access to `references/`, `static/`, `manifest.yaml`, scripts, assets, and `skills/nature-shared/`.

If you haven’t installed Claude Code yet:

```bash
npm install -g @anthropic-ai/claude-code
claude
```

First, clone the repository to a stable path:

```bash
mkdir -p ~/ai-skills
cd ~/ai-skills
git clone https://github.com/Yuan1z0825/nature-skills.git
```

Recommended approach: create a Claude Code subagent wrapper for commonly used skills. Example using `nature-reader`:

```bash
mkdir -p ~/.claude/agents
cat > ~/.claude/agents/nature-reader.md <<'EOF'
---
name: nature-reader
description: Use for Chinese-English paper reading, figure-aware translation, and source-grounded paper notes.
---

When invoked, first read `~/ai-skills/nature-skills/skills/nature-reader/SKILL.md` and follow it as the governing workflow.
Read supporting files from `~/ai-skills/nature-skills/skills/nature-reader/` and `~/ai-skills/nature-skills/skills/nature-shared/` only when needed.
Do not replace this skill with a generic paper-reading response.
EOF
```

Then start a new Claude Code session and request to use this subagent directly:

```text
Use the nature-reader subagent to turn this paper into a Chinese-English Markdown reader.
```

If you prefer slash commands, you can also create a command wrapper:

```bash
mkdir -p ~/.claude/commands
cat > ~/.claude/commands/nature-reader.md <<'EOF'
Read `~/ai-skills/nature-skills/skills/nature-reader/SKILL.md` first and follow it strictly.
Read directly needed supporting files from `~/ai-skills/nature-skills/skills/nature-reader/` and `~/ai-skills/nature-skills/skills/nature-shared/`.

$ARGUMENTS
EOF
```

Use it in Claude Code:

```text
/nature-reader Turn this paper into a complete side-by-side Chinese-English Markdown reader.
```

When installing other skills, replace `nature-reader` in the examples with the corresponding directory name, such as `nature-polishing`, `nature-writing`, `nature-reviewer`, `nature-response`, or `nature-figure`. To update later, simply run:

```bash
cd ~/ai-skills/nature-skills
git pull
```

As long as the wrapper still points to this stable clone path, there’s no need to repeatedly copy skill files.

**Automatic Updates (Optional)**

If you want Claude Code to automatically pull upstream updates at the start of each session, you can use `scripts/autoupdate-skills.sh` together with a `SessionStart` hook.

This method **directly copies** skills into `~/.claude/skills/` (which Claude Code automatically discovers and loads by directory name), instead of using the wrapper approach above. Choose one or the other.

First, keep a **dedicated** stable clone (used only for syncing skills; please do not make development commits in it):

```bash
mkdir -p ~/ai-skills
git clone https://github.com/Yuan1z0825/nature-skills.git ~/ai-skills/nature-skills
```

For the initial installation, copy the skills into Claude Code’s skills directory:

```bash
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --force
```

Then add a `SessionStart` hook in `~/.claude/settings.json` (if you already have `hooks`, merge this entry; do not completely overwrite):

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$HOME/ai-skills/nature-skills/scripts/autoupdate-skills.sh",
            "async": true,
            "timeout": 120
          }
        ]
      }
    ]
  }
}
```

`async: true` makes it run in the background without blocking startup. The script includes built-in protections: by default, it will not re-check the network within 6 hours; network failures or pull failures are silently skipped (`exit 0`, never stalling the session); it only resyncs when the upstream HEAD has genuinely changed; and it refuses to force-advance on a clone with uncommitted changes. The new version pulled will take effect in the **next** session (the skills for the current session are already loaded). Logs are written to `~/.local/state/nature-skills/autoupdate.log`.

Both the target directory and check frequency are configurable:

```bash
# Default sync to ~/.claude/skills; use --dest to point elsewhere, e.g., Codex:
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --dest ~/.codex/skills
# Only check at most once per hour:
~/ai-skills/nature-skills/scripts/autoupdate-skills.sh --throttle 3600
```

### 5.3 Installing for Codex

Use the repository’s bundled script to install or update Codex skills. The script synchronizes all top-level skill directories under `skills/` and performs a `diff` verification after copying; it will not overwrite other unrelated Codex skills.

```bash
git clone https://github.com/Yuan1z0825/nature-skills.git
cd nature-skills
scripts/update-codex-skills.sh --pull
```

If you have already cloned the repository:

```bash
cd nature-skills
scripts/update-codex-skills.sh --pull
```

Verify that the current Codex installation matches this checkout:

```bash
scripts/update-codex-skills.sh --check
```

If you use this script to update over the long term and want to clean up old skill directories that have been removed upstream:

```bash
scripts/update-codex-skills.sh --pull --prune
```

`--prune` will only delete directories previously recorded by this script that are no longer present in the current repository. On its first run, with no history, it will not guess and delete old directories.

You can also give the repository URL to Codex and let Codex run the install script. Recommended prompt:

```text
Please install Codex skills from this repository:
https://github.com/Yuan1z0825/nature-skills.git

Please clone the repository and then run scripts/update-codex-skills.sh --pull.
After installation, run scripts/update-codex-skills.sh --check to verify.
Please keep the complete skill directories under skills/; do not copy only SKILL.md.
```

If installing only a single skill, clearly specify the skill name:

```text
Only install nature-reader from this repository:
https://github.com/Yuan1z0825/nature-skills.git

If this skill needs shared files, please also install skills/nature-shared.
```

Key rule: keep the complete directory structure. Please copy or reference the entire skill folder, not just `SKILL.md`, because many skills rely on `references/`, `static/`, `manifest.yaml`, scripts, assets, or shared files.

The install script does not automatically install Python dependencies. Install them on demand when you need to use the relevant scripts or MCP services:

```bash
python -m pip install -r skills/nature-paper-to-patent/requirements.txt
python -m pip install -r skills/nature-paper-to-patent/scripts/disclosure/requirements-cnipa.txt  # Optional: CNIPA publication announcement search
python -m pip install -r skills/nature-academic-search/mcp-server/requirements.txt
```

If you enable the CNIPA publication announcement search for `nature-paper-to-patent`, you also need to run `python -m playwright install chromium`.

The MCP service for `nature-academic-search` additionally requires configuring `PUBMED_EMAIL` separately. Optional providers like Scopus / ScienceDirect need to be configured using local credentials; do not write API keys into repository files.

After installation, start a new Codex session and describe the task naturally, for example:

```text
Turn this paper into a complete side-by-side Chinese-English Markdown reader.
```

```text
Turn this paper into a Chinese PPT.
```

If you use open-source agent / programming frameworks like OpenClaw, OpenCode, or Hermes, see [OpenClaw / OpenCode / Hermes Integration Tutorial](docs/open-source-agent-frameworks.md).

**Automatic Updates (Optional)**

Codex supports a global `SessionStart` hook. After keeping a dedicated clone, you can check for updates and sync the new version to `~/.codex/skills/` each time a Codex session starts or resumes.

First, create the dedicated clone and perform the initial sync:

```bash
mkdir -p ~/.codex
git clone https://github.com/Yuan1z0825/nature-skills.git ~/.codex/.nature-skills-src
~/.codex/.nature-skills-src/scripts/autoupdate-skills.sh \
  --dest ~/.codex/skills --force
```

Then create or merge `~/.codex/hooks.json`:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "/bin/bash \"$HOME/.codex/.nature-skills-src/scripts/autoupdate-skills.sh\" --dest \"$HOME/.codex/skills\"",
            "timeout": 75,
            "statusMessage": "Checking Nature Skills updates"
          }
        ]
      }
    ]
  }
}
```

If `hooks.json` already contains other hooks, please merge the `SessionStart` entry; do not completely overwrite. After enabling or modifying hooks for the first time, run `/hooks` in Codex to inspect and trust it. Codex currently executes command hooks synchronously, so this approach relies on the script’s built-in 6-hour throttle, 60-second network protection, and automatic skip on network failure to avoid repeated network checks on every session or blocking startup due to update failures.

The update log is located at `~/.local/state/nature-skills/autoupdate.log`. New skills pulled usually take full effect in the next session.

## 7. Contributing & Development

### 7.1 Shared Design Principles

All skills adhere to the following principles:

1. **Prefer primary sources**: Rules are based on published Nature content, official journal guidelines, or well-defined local sources, not on vague aesthetic preferences.
2. **Explicit over implicit**: Every rule should state its rationale, not just an assertion.
3. **Section- and task-aware**: Academic writing, figures, citations, and responses all depend on context; different paper sections use different logic.
4. **Output-first**: Every skill should return a directly usable artifact, such as pasteable text, `.svg`, `.pptx`, `.docx`, or concrete suggestions.
5. **Extensible**: Each skill is self-contained in its own directory; adding a new skill should not require modifying existing skills.

### 7.2 Repository Directory Structure

```text
skills/
├── nature-shared/              # Must be preserved when skills reference ../nature-shared
├── nature-<topic>/
│   ├── README.md
│   ├── README_EN.md
│   ├── SKILL.md
│   ├── manifest.yaml     # Present in router-style skills
│   ├── static/           # Present in router-style skills
│   └── references/...
└── nature-proposal-writer/
    ├── README.md
    ├── README_EN.md
    ├── SKILL.md
    ├── scripts/...
    ├── templates/...
    └── references/...
```

### 7.3 New Skill Workflow

When adding a skill to this repository, follow this process:

**1. Create the skill directory**

```text
skills/nature-<topic>/
```

**2. Add required files**

| File | Required? | Purpose |
|------|----------|------|
| `SKILL.md` | Required | Frontmatter (`name`, `description`) + rules + workflow; loaded by agents when triggered |
| `README.md` | Required | Human-facing Chinese description page |
| `README_EN.md` | Required | English description page matching the Chinese detail page |
| `references/*.md` | Recommended for complex skills | Modular rule files, e.g., APIs, design theories, tutorials, chart types, etc. |

**3. Write the Chinese and English READMEs**

Every new skill must provide both `README.md` and `README_EN.md`. The README is the human-facing entry page for the skill; it is not a duplicate of `SKILL.md`, nor an installation manual. Its goal is to let users judge within 30 seconds: can this skill solve my problem, what should I feed it, what will it produce, and what are its boundaries.

Basic rules:

- The Chinese README and English README must mirror each other exactly: the number of headings, their order, and the information points must match; do not write the English page as a separate, independent template.
- The top of the page should contain the skill name, a language switch link, and a one-sentence positioning statement.
- Use the base structure below by default; only insert optional sections when truly necessary.
- Do not repeat the repository’s installation tutorial, author info, changelog, development process, full file tree, or large chunks of internal implementation details in a single skill’s README.
- Complex rules, API parameters, long tutorials, script descriptions, and template indexes should go into `references/`, `static/`, `scripts/`, or `SKILL.md`; the README keeps only signposts.
- If a skill has visual assets, a small preview table may be included; do not turn the README into a large gallery or lengthy technical manual.

Base structure for the Chinese README:

```markdown
# `nature-<topic>` 技能

[English](README_EN.md)

A one-sentence description of this skill's positioning, main tasks, and usage boundaries.

## 适合用它做什么
## 典型请求
## 你需要提供
## 产出
## 边界
## 相关技能
```

The English README must correspond as:

```markdown
# `nature-<topic>` Skill

[中文说明](README.md)

One sentence describing the skill's role, main task, and usage boundary.

## What To Use It For
## Typical Requests
## What You Need To Provide
## Outputs
## Boundaries
## Related Skills
```

Optional sections must be inserted synchronously in both Chinese and English and maintain the same order. Common optional sections include:

| Chinese Section | English Section | When to Use |
|----------|----------|----------|
| `## 工作方式` | `## Workflow` | When you need to explain the core process or routing method |
| `## 运行和依赖` | `## Runtime and Dependencies` | When there are scripts, MCP, API keys, local config, or external dependencies |
| `## 示例预览` | `## Example Preview` | When there are a few figures, screenshots, or visual assets worth showing |
| `## 内置参考` | `## Built-In References` | When you need to point to `references/`, `assets/`, or demos |
| `## 方法来源` | `## Method Sources` | When writing, review, or analysis rules come from specific sources |
| `## 三种模式` | `## Three Modes` | When the skill has clear compose/revise/hybrid modes |
| `## 与 ... 的关系` | `## Relationship With ...` | When it’s easily confused with another skill and needs a clarification of their division |

Before submitting, at least run these README checks:

```bash
python scripts/validate-readmes.py
git diff --check
for d in skills/nature-*; do
  [ -f "$d/README.md" ] && [ -f "$d/README_EN.md" ] || continue
  rg -q '^\[English\]\(README_EN\.md\)$' "$d/README.md"
  rg -q '^\[中文说明\]\(README\.md\)$' "$d/README_EN.md"
  test "$(rg -c '^## ' "$d/README.md")" = "$(rg -c '^## ' "$d/README_EN.md")"
done
```

**4. Record a usage tutorial**

When submitting a PR, please also record a short usage tutorial that explains what problem this skill solves, how to trigger it, what input it needs, and what output it produces. You can attach the video, a screen recording link, or a publicly accessible tutorial URL in the PR description.

**5. Configure `SKILL.md` frontmatter**

```yaml
---
name: nature-<topic>
description: >-
  One sentence describing what this skill does, when it triggers, main output format, and core usage scenarios.
---
```

**6. Update the skill index**

Add a row to the [Skill Index](#6-skill-index) table above:

```markdown
| [`nature-<topic>`](skills/nature-<topic>/README.md) | Draft / Stable | One-sentence purpose | Trigger words | [Details](skills/nature-<topic>/README.md) |
```

**7. Set the status label**

| Label | Meaning |
|-------|------|
| `Draft` | Rules defined, but not yet tested on real cases |
| `Beta` | Tested on examples, boundary issues may still exist |
| `Stable` | Validated on real academic content, rules relatively stable |
