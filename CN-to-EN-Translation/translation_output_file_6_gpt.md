# Act I: Generation (November 2022 – 2023)

On **November 30, 2022**, ChatGPT was launched. Within roughly two months, it reached around **100 million monthly active users**.

What truly changed, however, was not natural language processing itself—GPT-3 had already existed for two years—but rather the **interaction paradigm**. Before ChatGPT, large language models were APIs that only engineers could effectively use. After ChatGPT, they became conversational interfaces accessible to everyone. 

The central limitation during this era was straightforward:

> **The model could generate—but it could not act.**

It could draft an email, but it could not send it.

It could write code, but it could not execute it.

The relationship between user and model was fundamentally a stateless, single-turn exchange: **you ask, I answer**. 

The engineering discipline born from this period was **Prompt Engineering**—learning how to phrase prompts so that the model would produce better outputs.

Techniques such as:

* Few-shot prompting
* Chain-of-thought prompting
* Role prompting

all attempted to solve the same problem:

> maximize information density within the limited space of a single API call. 

---

# Act II: Connection (2023–2024)

March 2023 marked another major inflection point.

OpenAI released **GPT-4**, introducing multimodal capabilities and substantially longer context windows.

That same month, **ChatGPT Plugins** debuted, allowing models to interact with external APIs and access live data for the first time. The model had, metaphorically speaking, **grown hands**.

Three months later, in June, OpenAI introduced the **Function Calling API**, standardizing tool invocation through structured JSON outputs generated by the model. 

This represented a crucial evolutionary step.

The model progressed from merely **being able to speak** to **being able to connect**.

However, the Plugin ecosystem quickly revealed its weaknesses.

Each plugin required:

* its own OAuth flow,
* its own schema definition,
* its own error-handling implementation.

Connecting ten plugins was already painful.

Connecting one hundred was effectively impossible. 

During the second half of 2023, **LangChain** rose to prominence by introducing an abstraction layer intended to solve the "connection problem."

It undeniably lowered the barrier to entry.

But it also introduced the familiar costs of excessive abstraction:

* too many layers,
* difficult debugging,
* unpredictable performance.

Around the same time, projects such as **AutoGPT** and **BabyAGI** attempted to build autonomous execution loops in which models could repeatedly plan and act on their own.

Most of these projects quickly faded after impressive demonstrations because they lacked reliable stopping criteria and robust validation mechanisms. 

> **Lesson One**
>
> Enabling a model to connect to tools is necessary—but it is far from sufficient.
>
> **Connection is not orchestration, and orchestration is not governance.** 

# Act III: Reasoning (2024)

The year **2024** was the year that **reasoning models** took center stage.

In September, OpenAI introduced the **o1** series, whose defining characteristic was the ability to *"spend more time thinking before responding."* This represented a qualitative leap in performance on mathematics and programming tasks.

Three months later, in December, the ARC Prize announced that **OpenAI o3** had achieved **87.5%** on the **ARC-AGI-1 Semi-Private Evaluation** under a high-compute configuration, astonishing the AI community.

At the same time:

* Anthropic's **Claude 3.5 Sonnet** demonstrated exceptional performance in code generation and long-document comprehension.
* **DeepSeek-R1**, as an open-weight model, showed that high-performance reasoning was no longer exclusive to proprietary systems. 

More importantly, two events at the end of 2024 fundamentally reshaped the engineering landscape. 

---

## First: Anthropic Introduces the Model Context Protocol (MCP)

Anthropic released the **Model Context Protocol (MCP)**.

This was **not** simply another plugin framework.

Instead, it was an **open standard** based on **JSON-RPC 2.0** that defined how AI applications should communicate with external tools and data sources.

Its central insight was deceptively simple:

The connection problem is fundamentally an **N × M problem**.

* N AI applications
* M external tools or data sources

Without a common protocol, every application needs a custom integration for every tool.

In other words:

> **N applications × M tools = N × M connectors.**

MCP reduces this to:

> **N + M**

Each AI application implements the MCP client once.

Each tool implements an MCP server once.

The ecosystem can then interoperate through the protocol itself.

OpenAI and Google DeepMind later announced support for MCP, and in **December 2025**, Anthropic donated the protocol to the **Agentic AI Foundation** under the **Linux Foundation**. 

---

## Second: *Building Effective Agents*

In **December 2024**, Anthropic published **Building Effective Agents**.

For the first time, the article systematically described engineering patterns for AI agents—from the simplest prompt chaining workflows to evaluator–optimizer loops.

More importantly, it articulated a design principle that would later become one of the foundations of Harness Engineering:

> **Start with the simplest possible pattern. Introduce additional structure only when the added complexity demonstrably produces better outcomes.** 

---

By **2025**, Anthropic had further elevated **Context Engineering** into a distinct engineering discipline through **Effective Context Engineering for AI Agents**.

The emphasis shifted.

The difficult question was no longer:

> *How should we write prompts?*

Instead, it became:

> *At each step, what information should be provided to the model, in what form, and at what moment?*

This represented the crucial transition layer between **Prompt Engineering** and **Harness Engineering**.

The engineering abstraction had moved:

* from a **single model invocation**,
* to **per-step context management**,

but it had not yet advanced to managing the **entire outer execution loop** of a long-running task. 

---

> **Lesson Two**
>
> Better reasoning improves the quality of **individual steps**.
>
> It does **not** automatically make long-running tasks reliable.
>
> A model capable of solving International Mathematical Olympiad gold-medal problems can still spend four hours building a full-stack application and eventually forget what it was trying to accomplish. 

---

# Act IV: Action (2025)

If **2023** was the year of the chatbot, and **2024** the year of multimodal AI, then **2025** became the **Year of the Agent**. 

The year began with the open-source release of **DeepSeek-R1**, prompting the industry to reassess the competitive landscape of foundation models.

What followed was an explosion of agent products:

* Claude Code — a coding agent inside the terminal
* GitHub Copilot Agent Mode
* Cursor's autonomous coding loops
* Manus — a browser automation agent
* OpenAI Operator

Meanwhile, the MCP ecosystem expanded rapidly.

The community built **thousands** of MCP servers, while SDKs became available for virtually every major programming language.

Google also introduced the **Agent2Agent (A2A)** protocol to enable interoperability between agents developed by different vendors. 

Yet the most important discovery of 2025 was **not what agents could accomplish**.

It was **how they failed**. 

Typical failure patterns included:

* attempting to complete an entire task in one uninterrupted run, only to exhaust the context window halfway through;
* declaring a task "fully completed" after reaching roughly 70% progress;
* cascading failures in multi-agent systems, where a minor mistake propagated into a complex, difficult-to-debug system-wide failure;
* repositories gradually accumulating severe **AI slop**—redundant code, inconsistent naming conventions, stale documentation, and architectural drift.

These were **not intelligence problems**.

They were **systems engineering problems**. 

---

> **Lesson Three**
>
> Agents have already reached the point where they can work autonomously for hours.
>
> The engineering infrastructure surrounding them, however, remains largely designed for the era of **single conversations**.
>
> This mismatch is precisely why **Harness Engineering** emerged. 

# Act V: Governance (2026 – Present)

By the beginning of **2026**, the industry's focus had begun to shift.

The central question was no longer:

> **How can we make agents more capable?**

Instead, it became:

> **How can we keep agents from going off the rails?** 

The term **Harness Engineering** did not suddenly appear overnight. Rather, it rapidly entered mainstream technical discourse beginning in **February 2026**. 

### February 5, 2026

Mitchell Hashimoto published **My AI Adoption Journey**, explicitly using the phrase:

> **"Engineer the Harness."**

This is widely regarded as one of the moments when the term entered mainstream discussion. 

### February 11, 2026

OpenAI published an engineering article titled:

> **Harness Engineering: Leveraging Codex in an Agent-First World**

In it, the company described how a small team built an internal beta product over five months starting from an empty repository.

Publicly, OpenAI described the project as involving **"zero lines of manually handwritten code."**

The repository eventually grew to roughly **one million lines of code**, generating approximately **1,500 pull requests**.

A more precise description is that the initial scaffold was still generated by Codex using a small amount of template guidance. After that, the application logic, tests, CI pipelines, documentation, observability infrastructure, and internal tooling were produced as much as possible by AI agents.

The key insight was that the engineer's primary responsibility had shifted toward:

* designing environments,
* specifying intent,
* and building feedback loops. 

### March 2026

Anthropic published **Harness Design for Long-Running Application Development**.

Its earlier two-role architecture—**initializer** and **coding agent**—was upgraded into a three-role system:

* Planner
* Generator
* Evaluator

The work demonstrated that evaluators continued to produce meaningful quality improvements, even near the frontier of model capability. 

### April 2026

The Thoughtworks / Martin Fowler community further systematized the concept in **Harness Engineering – First Thoughts**.

Their framework organized harnesses around two complementary mechanisms:

* **Guides** (feed-forward control)
* **Sensors** (feedback control)

Each was further divided into:

* **Computational** (deterministic)
* **Inferential** (LLM-based or probabilistic)

Together they formed a **2×2 control matrix**.

Accordingly, April is better understood not as the moment the term was coined, but as the point at which the methodology itself became conceptually complete.

# What Exactly Is a Harness?

Rather than beginning with a definition, let's derive the concept from first principles. 

---

# The Five Fundamental Challenges of Agents

An agent is fundamentally:

> **A system that autonomously pursues goals in an open environment.**

Embedded within this definition are **five engineering challenges**, none of which can be solved merely by building a smarter model. 

## 1. Persistent State

**The challenge**

An agent must remember what it has done across time and across sessions.

**Why models alone cannot solve it**

Foundation models are inherently stateless.

Context windows are finite.

Neither naturally supports long-term continuity. 

---

## 2. Goal Alignment

**The challenge**

During long-running tasks, agents are prone to drifting away from their objectives, entertaining themselves with irrelevant work, or prematurely declaring success.

**Why models alone cannot solve it**

Models lack an external anchor that continually recalibrates what "done" actually means. 

---

## 3. Verifiable Actions

**The challenge**

Every action taken by an agent is probabilistic.

A system must distinguish between:

* performing an action,
* and performing the correct action.

**Why models alone cannot solve it**

Models exhibit an inherent tendency toward overly optimistic self-evaluation and are poor judges of their own outputs. 

---

## 4. Entropy Suppression

**The challenge**

Continuous autonomous generation inevitably accumulates redundancy, inconsistency, and drift.

**Why models alone cannot solve it**

Models naturally imitate existing patterns—even when those patterns are themselves poor or low quality. 

---

## 5. Human–Agent Boundaries

**The challenge**

A system must explicitly define:

* when an agent should continue autonomously,
* and when control should return to a human.

**Why models alone cannot solve it**

Models possess no consistently reliable awareness of their own uncertainty.

They cannot robustly determine when they should stop and hand control back to a person. 

---

Harness Engineering is the systematic engineering discipline devoted to addressing these five challenges. 

---

# A Precise Definition

Anthropic provides a particularly useful definition in **Demystifying Evals for AI Agents**:

> An **agent harness** (or **scaffold**) is the system that enables a model to function as an agent. It is responsible for handling inputs, orchestrating tool calls, and returning results.

More importantly, Anthropic argues that when evaluating an AI agent, we are **not evaluating the model in isolation**.

We are evaluating:

> **Model + Harness**

This shifts the unit of analysis away from model parameters alone and toward the surrounding execution loop in which the model operates. 

The article also emphasizes an important distinction that is frequently overlooked.

An **agent harness** is **not** the same thing as an **evaluation harness**.

The former is responsible for runtime orchestration:

* processing inputs,
* coordinating tools,
* managing state.

The latter exists for benchmarking:

* executing batches of tasks,
* recording trajectories,
* running graders,
* aggregating evaluation metrics.

Many discussions conflate these two ideas.

Throughout this article, **Harness Engineering** refers specifically to the **runtime outer-loop system**. 

From this perspective, we can formulate a more precise definition:

> **Harness = the outer-loop system that enables a model to operate as an autonomous agent.** 

It encompasses:

* task planning and decomposition,
* durable state,
* tool orchestration,
* verification gates,
* feedback loops,
* rollback mechanisms,
* human handoff points,
* and audit logs.

Consequently, evaluating an AI agent means evaluating **the combined performance of the model and its harness**, rather than the model alone. 

## Several Key Points Are Worth Expanding

The definition above contains several implications that deserve closer examination. 

### The Outer Loop Is the Key Concept

A model's reasoning process is the **inner loop**: given a context, it generates the next step.

A harness constitutes the **outer loop**:

* deciding when to start a new reasoning cycle,
* determining what context to provide,
* validating the model's outputs,
* deciding when to roll back,
* and determining when execution should terminate.

The quality of the inner loop depends primarily on the model itself.

The quality of the outer loop depends on the design of the harness. 

### A Harness Is Not an Upgraded Prompt

A single prompt cannot solve problems such as:

* cross-session state,
* validation gates,
* tool discovery,
* failure recovery,
* or continuous entropy control.

Treating a harness as merely "a much longer system prompt" is currently one of the most common failure modes in agent engineering. 

### A Harness Is Not a Framework

LangChain is a framework.

CrewAI is a framework.

Harness Engineering is **not**.

It is an engineering practice in the same way that **DevOps** is not a software product but an engineering culture and methodology. 

---

# Three Layers of Engineering Abstraction

> **Prompt → Context → Harness** 

To understand where Harness Engineering fits, we must first understand its relationship with the two abstraction layers beneath it.

These layers are **not replacements for one another**. Instead, they represent progressively higher levels of abstraction.

### Prompt Engineering

Focuses on:

> **How should a single prompt be written?**

Its scope is limited to one model invocation.

---

### Context Engineering

Focuses on:

> **What information should the model receive at each step?**

This extends beyond prompt wording to include retrieval, memory, tool outputs, and contextual organization.

---

### Harness Engineering

Focuses on:

> **How should the entire execution pipeline operate?**

It governs the complete lifecycle of an agent:

* multiple reasoning steps,
* persistent state,
* tool orchestration,
* verification,
* recovery,
* and human intervention.

---

### 📌 Key Insight

**Context Engineering** answers the question:

> *What should the model be fed at each step?*

**Harness Engineering** answers the question:

> *How should the entire pipeline run?*

Context Engineering is therefore a **subset** of Harness Engineering.

From the perspective of application developers, Harness Engineering can be viewed as a specialized form of Context Engineering.

However, a harness also encompasses:

* multi-step execution structures,
* tool mediation,
* verification gates,
* and durable state,

all of which extend beyond the scope of per-step context management. 

---

# The Six Core Engineering Components of a Harness

This is the most technically intensive section of the article.

For each component, we will examine:

* the problem it addresses,
* Anthropic's and OpenAI's engineering approaches,
* and the underlying design principles.

# 1. Durable State Surfaces: Preventing Agent Amnesia

### The Problem

The central challenge of long-running agents resembles a software engineer who completely loses their memory every time they start a new shift.

Because context windows are finite, complex projects cannot be completed within a single session.

Agents therefore require a mechanism for bridging multiple sessions. 

### Anthropic's Solution

Rather than attempting to build an infinitely long context window, Anthropic externalizes state into persistent artifacts that can survive across sessions.

Their initializer agent prepares the environment by creating:

* **init.sh** — the startup script
* **claude-progress.txt** — a structured progress log
* an initial Git commit serving as the baseline snapshot

It also generates a **feature list** that expands high-level requirements into more than 200 concrete features, each initially marked as **failing**.

Subsequent coding agents work only on incremental improvements.

At the end of each session, they leave behind:

* structured progress updates,
* and a clean repository state.

One critical rule governs the process:

> Agents may update only the pass/fail status of features.

> They are **not** allowed to modify the test definitions themselves. 

Although this feature-list design appears almost simplistic, it is remarkably effective.

It transforms the definition of "done" from an agent's subjective impression into an **external, durable, structured representation of completion**.

An agent no longer needs to remember what happened previously.

It only needs to read:

* the feature list,
* and the Git diff,

allowing it to resume work within roughly thirty seconds. 

Anthropic later uncovered an even deeper issue:

## Context Anxiety

Even after implementing conversation compaction through summarization, agents still exhibited degraded performance simply because they perceived the context as becoming "too full."

The solution was **not** better summarization.

Instead, Anthropic introduced **context reset**.

Rather than continuing the existing conversation, a completely new agent starts with a fresh context.

All necessary information is transferred through external state artifacts rather than conversation history.

This approach is more radical than compaction—but proved significantly more effective. 

---

### 📌 Design Principle

State is **not** synonymous with saving chat history.

True **durable state** consists of structured artifacts that enable an agent to:

* cold start,
* read the current project state,
* understand where work previously stopped,
* and continue execution,

without relying on any previous conversation.

If your agent cannot determine within thirty seconds:

* what has already been completed,
* and what should happen next,

then your state management strategy has failed. 

# 2. Decomposition & Plans: Breaking Long Tasks into Manageable Units

### The Problem

Tell an agent:

> **"Build a clone of Claude.ai."**

Its instinct is often to tackle the entire project in a single session.

The result is predictable:

* the context window overflows,
* or the agent reaches the halfway point and confidently declares the task "finished." 

---

## The Evolution

### November 2025

Anthropic initially addressed this problem using a two-role architecture:

* **Initializer** — responsible for planning and project initialization.
* **Coding Agent** — responsible for implementing features incrementally. 

---

### March 2026

The architecture evolved into a three-role system:

* **Planner**
* **Generator**
* **Evaluator** 

Each role has a clearly defined responsibility.

### Planner

The planner does **not** write code.

Instead, it expands a one- or two-sentence product idea into:

* a complete product specification,
* and a step-by-step feature list.

### Generator

The generator implements one feature at a time.

After completing each feature, it creates a commit.

### Evaluator

The evaluator independently reviews the generator's work.

It determines whether each feature:

* passes,
* fails,

and provides concrete recommendations for improvement.

---

OpenAI follows a similar philosophy through project artifacts such as:

* **PLANS.md**
* **Implement.md**
* **Documentation.md**

Complex work is planned first.

Execution proceeds milestone by milestone.

Every milestone includes validation.

Meanwhile, the documentation is continuously updated to serve as shared project memory. 

---

### 📌 Design Principle

A plan must be elevated to a **first-class artifact**.

It should not exist merely as part of a conversation.

Instead, it should:

* live in the repository,
* be version controlled,
* remain readable by future agents,
* and be referenced by validation gates.

A plan that exists only in chat is not truly a plan.

It is merely an idea. 

---

# 3. Feedback Loops: Guides and Sensors

### The Problem

Suppose an agent writes some code.

How do we determine whether it is correct?

Can we simply ask the agent to evaluate its own work?

Anthropic discovered an uncomfortable truth:

> When asked to judge its own output, an agent tends to praise itself enthusiastically—even when human reviewers consider the quality mediocre. 

This creates the need for a feedback system that does **not** depend on self-evaluation.

By April 2026, the community had converged on a particularly elegant abstraction:

A harness can be understood as a **2 × 2 control matrix**. 

---

## Guides

Guides operate **before** the agent acts.

Their purpose is to increase the probability that the agent gets things right on the first attempt.

---

## Sensors

Sensors operate **after** the agent acts.

Their purpose is to detect mistakes and support self-correction.

---

Each category can be further divided into two forms of control:

### Computational Control

Characteristics:

* deterministic,
* inexpensive,
* fast.

Suitable for execution on every change.

Examples include:

* unit tests,
* type checking,
* linters,
* CI pipelines.

---

### Inferential Control

Characteristics:

* probabilistic,
* slower,
* more expensive.

Appropriate for subjective questions such as:

* Is this UI aesthetically pleasing?
* Is this architecture elegant?
* Is this code style appropriate?

These judgments generally require another language model or human-like evaluator. 

---

## Key Insights

Having **guides without sensors** means:

> The agent follows rules but never learns whether those rules actually worked.

Having **sensors without guides** means:

> The agent repeatedly makes the same mistakes before being corrected.

Computational controls are:

* cheap,
* fast,
* deterministic,

making them ideal for routine validation.

Inferential controls are:

* slower,
* more costly,
* non-deterministic,

but indispensable for evaluating subjective quality. 

Anthropic's **Evaluator–Optimizer** pattern fits this framework almost perfectly.

At the same time, Anthropic acknowledges an important nuance:

Evaluators are **not always necessary**.

Once the underlying model surpasses a certain capability threshold, evaluators gradually shift from being an essential component to an optional optimization whose cost may outweigh its benefit.

This illustrates an important principle:

A good harness is **not** a rigid template.

It is an adaptable system that evolves alongside the capabilities—and limitations—of the underlying model. 

# 4. Legibility: Building a Perception Surface for Agents

### The Problem

An agent can generate code.

But can it **see** the software it has just built?

Can it interpret runtime error logs?

Can it understand performance metrics?

Without access to these signals, an agent is effectively working blind. 

OpenAI articulated this idea with a particularly sharp observation:

> **Any knowledge that is not visible to an agent during execution effectively does not exist.** 

This is not merely rhetorical.

To improve an agent's situational awareness, OpenAI implemented several concrete engineering practices:

* Launch a dedicated browser instance for every Git worktree so the agent can **see** the application's user interface via the **Chrome DevTools Protocol (CDP)**.
* Expose logs, metrics, and execution traces directly to the agent for inspection.
* Treat the repository itself as the **system of record**, storing:

  * design principles,
  * product intent,
  * implementation plans,
  * known technical debt,
  * and Architecture Decision Records (ADRs).
* Keep **AGENTS.md**, structured documentation, execution plans, and knowledge documents under version control inside the repository so they become part of the agent's observable environment. 

OpenAI also cautions against a common anti-pattern.

An excessively large **AGENTS.md** inevitably decays over time.

It consumes valuable context window space while diluting the importance of every instruction.

A better approach is to treat **AGENTS.md** as an index, with the substantive knowledge distributed across structured documentation. 

---

### 📌 Design Principle

Legibility is **not** about making code more elegant.

It is about making the following artifacts visible to the agent:

* organizational knowledge,
* engineering constraints,
* acceptance criteria,
* decision history.

This reframes knowledge management itself.

Instead of being merely a collaboration problem among humans, it becomes an execution problem for autonomous agents.

From an agent's perspective:

* tribal knowledge shared in Slack,
* architectural constraints communicated verbally,
* documentation scattered across external systems,

all effectively **do not exist** unless they are represented as runtime-accessible artifacts. 

---

# 5. Tool Mediation: The More Tools an Agent Has, the More It Needs a Harness

### The Problem

As the MCP ecosystem expands, a single agent may connect to:

* dozens of MCP servers,
* hundreds of tools.

Simply loading every tool definition into the model's context quickly becomes problematic.

The consequences include:

* dramatically higher token consumption,
* increased latency,
* and agents becoming lost in an overwhelming sea of available tools. 

Anthropic's work on MCP and code execution introduced an important engineering principle:

> **Don't let the model call tools directly.**
>
> **Let the model write code that calls the tools instead.** 

---

## Direct Tool Calling

The traditional workflow looks like this:

1. Every tool definition is loaded into the model's context.
2. The model selects a tool.
3. The tool is invoked.
4. Results are returned to the context.
5. The model continues reasoning.

Every intermediate step consumes valuable context space. 

---

## Code Execution

The alternative workflow is fundamentally different:

1. The model writes a piece of code.
2. The code executes inside a sandbox.
3. The execution environment discovers and invokes MCP tools as needed.
4. Data filtering and intermediate processing occur entirely outside the language model.
5. Only the final result is returned to the model.

Tool discovery, orchestration, filtering, and execution all happen outside the model's reasoning loop. 

The underlying idea is profound.

Tool usage is moved out of the model's **inner loop** and into a more efficient **external execution loop**.

This is precisely what Harness Engineering is about.

A harness is **not** merely a registry of tools.

It is the system-level design that determines:

* how tools are discovered,
* when they become available,
* at what level of abstraction they are exposed,
* whether intermediate results enter the model's context,
* where state is stored,
* and how failures are recovered from. 

---

# 6. Entropy Control: Continuous Garbage Collection for Agents

### The Problem

Fully autonomous agents inevitably reinforce existing patterns within a codebase—even when those patterns are inconsistent, suboptimal, or simply poor.

Over time, architectural drift and entropy become unavoidable. 

OpenAI describes the issue candidly.

Initially, engineers spent roughly **20% of their time every week** cleaning up **AI slop**:

* redundant code,
* outdated documentation,
* inconsistent naming,
* dead copy-and-pasted code.

Eventually, this cleanup process itself became automated. 

Examples include:

* **Documentation consistency agents** that verify documentation matches the code.
* **Refactoring agents** that systematically eliminate technical debt.
* **Architectural enforcement** implemented through CI to mechanically preserve module boundaries. 

---

### 📌 Design Principle

A harness is responsible for far more than simply enabling an agent to run.

It must also continuously suppress the systemic noise that autonomous agents naturally amplify over time.

This is the fundamental distinction between a **harness** and a conventional **agent framework**.

A framework focuses on startup and orchestration.

A harness focuses on **long-term maintainability, governance, and system health**. 

# Harnessability: Not Every System Is Equally Amenable to a Harness

If Harness Engineering is understood merely as "adding more rules and feedback loops to an agent," we have not gone deep enough.

A more fundamental question is:

> **Not every system is equally easy to harness.** 

OpenAI's engineering practice repeatedly points to the same conclusion.

Their ability to scale Codex to extremely high throughput was not solely a consequence of stronger models.

It was equally the result of continuously:

* pushing knowledge back into the repository,
* turning plans into durable artifacts,
* versioning architectural decisions,
* and making the development environment increasingly legible to agents. 

This leads to a compelling observation:

Some systems are **inherently more harnessable** than others.

Systems characterized by:

* strong typing,
* comprehensive test coverage,
* well-defined module boundaries,
* version-controlled documentation,
* and observable runtime behavior,

naturally exhibit higher harnessability.

By contrast, systems whose knowledge is scattered across:

* individual engineers' memories,
* chat applications,
* hallway conversations,
* and undocumented tribal knowledge,

will eventually hit the same wall—even with increasingly capable models:

> **If the agent cannot see it, it cannot understand it.**
>
> **If it cannot understand it, it cannot govern it.** 

Consequently, in the age of AI agents, engineering infrastructure itself becomes a primary determinant of success.

The quality of a team's:

* CI pipelines,
* structured documentation,
* architectural boundaries,

is no longer simply a matter of engineering maturity.

It directly determines how far autonomous agents can operate effectively within that system.

**Harnessability** is therefore poised to become one of the defining dimensions of **agent readiness**.

# Intent Systems

## A Deeper Paradigm Shift: From Instruction-Driven to Intent-Driven Computing

Everything discussed so far concerns the engineering components of a harness.

Yet if we focus solely on those components, we risk reducing Harness Engineering to a collection of implementation techniques.

A more fundamental transformation is taking place.

Harness Engineering is not merely another engineering practice.

It is the product of a broader **paradigm shift**. 

---

# Four Fundamental Shifts in Human–Computer Interaction

Viewed across the entire history of computing, interaction between humans and machines has undergone four major discontinuities. 

## 1. CLI (Command-Line Interface)

Humans were required to speak the machine's language precisely.

For example:

```bash
ls -la | grep .py
```

A single incorrect character caused the command to fail.

The underlying assumption was:

> **Humans adapt to machines.** 

---

## 2. GUI (Graphical User Interface)

Machines lowered the barrier to use through visual metaphors:

* folders,
* desktops,
* drag-and-drop interactions.

The assumption became:

> **Machines present themselves using metaphors humans naturally understand.** 

---

## 3. Apps (Mobile Applications)

Application logic became encapsulated within fixed interfaces.

Each feature corresponded to:

* a button,
* a screen,
* a predefined workflow.

The governing assumption was:

> **Humans choose among predefined paths.** 

---

## 4. Agents (Intent-Driven Systems)

Humans no longer specify every step.

Instead, they express a goal.

The system plans and executes autonomously.

The underlying assumption becomes:

> **Machines understand human intent and determine how best to accomplish it.** 

Every one of these transitions represents more than technological progress.

Each fundamentally redistributes **control**.

During the CLI era, humans retained virtually complete execution control.

In the Agent era, humans relinquish much of that control, retaining primarily:

* goal definition,
* and key decision points. 

---

## The Engineering Consequence

In an instruction-driven world, software bugs arise because:

> **The system failed to execute my instructions correctly.**

Traditional testing methodologies are well suited to this problem.

In an intent-driven world, however, failures become fundamentally different.

They arise because:

* the system misunderstood the user's intent,
* or correctly understood the intent but selected a poor execution strategy.

These failures require entirely new mechanisms for:

* verification,
* constraint,
* feedback,
* and governance.

These mechanisms are precisely what Harness Engineering is designed to provide. 

# Applications Are Becoming "CLI-ified"—But Not for Humans

One of the most counterintuitive trends of the Agent era is this:

Applications and websites are once again becoming **CLI-like**.

But this transformation is **not** intended for human users.

It is happening from the perspective of **AI agents**. 

The **Model Context Protocol (MCP)** is essentially the protocol-layer realization of this idea.

When an agent needs to interact with Google Drive, it should not have to:

* open a webpage,
* click buttons,
* navigate menus.

Instead, it should invoke structured programmatic interfaces.

An MCP server abstracts Google Drive into callable functions such as:

* `gdrive.getDocument`
* `gdrive.createFile`
* `gdrive.search` 

This shift has three important consequences. 

---

## First: The Meaning of Readability Has Changed

Traditionally, readability was designed for humans:

* intuitive user interfaces,
* clear information architecture,
* visually coherent layouts.

In the Agent era, readability must first serve machines.

Applications now need:

* structured APIs,
* machine-readable documentation,
* programmable permission models. 

---

## Second: Application Boundaries Are Dissolving

When agents invoke arbitrary tools through MCP and collaborate through protocols such as **Agent2Agent (A2A)**, applications cease to be destinations.

Instead, they become infrastructure.

Users no longer think:

> "Open an application to accomplish a task."

Instead, they simply express an intention, while the agent orchestrates multiple services behind the scenes. 

---

## Third: Harness Becomes the New Operating-System Layer

During the GUI era, operating systems managed:

* windows,
* files,
* processes.

In the Agent era, an equivalent layer must manage:

* agent lifecycles,
* tool discovery and authorization,
* context scheduling and garbage collection,
* multi-agent collaboration and isolation,
* points of human approval and intervention. 

---

# From Chatbots to AgentOS

Connecting all these threads reveals a clear evolutionary trajectory.

These stages do not simply accumulate features.

Each represents a fundamental shift in engineering abstraction. 

---

## Level 1: Chatbots (2022–2023)

Characteristics:

* single-turn conversations,
* stateless interactions,
* humans fully in the loop.

Primary value:

* information retrieval,
* content generation.

Dominant engineering abstraction:

> **Prompt Engineering**

Representative products:

* ChatGPT
* early versions of Claude

### Ceiling

Chatbots could **talk**, but they could not **act**.

Every conversation began from scratch. 

---

## Level 2: AI Browsers / Agent IDEs (2024–2025)

Characteristics:

* multi-step task execution,
* tool invocation,
* limited autonomy.

Primary value:

* workflow automation,
* task execution.

Dominant engineering abstraction:

> **Context Engineering + Lightweight Harnesses**

Representative products include:

* Claude Code,
* Cursor,
* Manus,
* Codex.

### Ceiling

Individual agents became increasingly capable.

However:

* long-running tasks remained unstable,
* multi-agent collaboration lacked standardized mechanisms,
* state management was still largely manual. 

---

## Level 3: AgentOS (2026–, Emerging)

The author deliberately adopts a cautious tone here.

**AgentOS is not yet a mature industry paradigm.**

Nevertheless, it has entered the research agenda.

For example:

* the 2024 **AIOS** paper proposed moving scheduling, context management, memory management, and access control from the application layer into an operating-system-like kernel;
* **ASPLOS 2026** introduced a dedicated **AgenticOS Workshop** focused on operating-system primitives for agent workloads. 

A more accurate statement is therefore:

> Harness Engineering is steadily pushing AI engineering problems upward—from the application layer toward the systems layer. 

When agents evolve beyond coding assistants into always-on, long-lived entities that cooperate across tools, identities, and organizations, user-space harnesses inevitably encounter operating-system-level challenges.

These include:

### Agent Lifecycle Management

Managing:

* initialization,
* execution,
* suspension,
* resumption,
* termination.

Agents are no longer stateless function calls.

They increasingly resemble long-running processes. 

---

### Context Scheduling

Context windows are scarce resources.

A system must determine:

* what information to load,
* when to compress it,
* and when to discard it.

This is effectively **memory management for agents**. 

---

### Multi-Agent Isolation and Collaboration

One agent's actions should not contaminate another's execution environment.

At the same time, they must share selected pieces of state.

This requires mechanisms analogous to:

* process isolation,
* inter-process communication (IPC). 

---

### Governance and Auditability

Every decision made by an agent should be traceable.

In domains such as:

* finance,
* healthcare,

this is not an optional feature.

It is a regulatory necessity. 

---

## 📌 Positioning

Harness occupies the **user-space layer** of AgentOS.

AgentOS itself functions as the kernel.

It manages:

* scheduling,
* isolation,
* resource allocation.

Harness operates like the shell and long-running daemons of a traditional operating system.

It manages:

* task decomposition,
* durable state,
* verification,
* feedback,
* and human handoffs.

The two concepts are therefore **complementary rather than competing**.

They naturally occupy different layers of the emerging software stack. 

# Five Common Symptoms

Having covered the theory, let's return to practice.

If you examine today's AI ecosystem, you'll find that most so-called **agent systems** are still held together by ad hoc integrations rather than coherent engineering.

The following five symptoms are particularly common. 

---

## Symptom 1: A Jungle of Frameworks

LangChain.

CrewAI.

AutoGen.

Agno.

n8n.

Each framework solves a small portion of the overall problem.

None, however, provides a complete lifecycle encompassing:

* planning,
* validation,
* rollback,
* and auditability.

Teams therefore end up stitching together multiple frameworks, producing fragile pipelines instead of governable systems. 

---

## Symptom 2: A Chatbot Shell Wrapped Around an Agent Core

Many products are, in essence, chatbot interfaces enclosing an agent execution loop.

They often lack:

* genuine state management,
* task decomposition,
* verification gates.

As a result, they look impressive in demonstrations but fail repeatedly in production environments. 

---

## Symptom 3: Tool Registration Is Not Tool Governance

MCP makes connecting tools remarkably easy.

But:

> **Being able to connect tools is not the same as being able to use them well.**

When an agent is presented with fifty tools simultaneously, it may:

* become confused,
* invoke redundant tools,
* choose inefficient execution paths.

Some engineering teams have even found that exposing **every available tool** initially degraded performance.

Only after reducing the available tools to the **minimum necessary set** did performance improve. 

---

## Symptom 4: One-Off Rules Instead of Evolvable Constraints

Many teams configure their agents using a massive **AGENTS.md** file or an enormous system prompt.

Experience suggests this approach is destined to fail.

When **everything is important**, nothing truly stands out.

Agents perform local pattern matching rather than consciously navigating an evolving body of knowledge.

Moreover, engineering rules decay faster than humans can maintain them. 

---

## Symptom 5: Missing the "On-the-Loop" Mindset

There is an important distinction between:

**In the loop**

When an agent produces an unsatisfactory result, a human manually edits the artifact.

**On the loop**

Instead of correcting the output directly, the engineer improves the **harness**, so the system produces better results automatically the next time.

Most teams remain trapped in the **in-the-loop** mindset.

They fix individual mistakes instead of systematically improving the control loops that generated those mistakes. 

---

# What Harness Engineering Is Not

Clarifying the boundaries of a concept is just as important as defining it. 

Harness Engineering is **not**:

### A Longer System Prompt

A prompt alone cannot manage:

* cross-session state,
* validation gates,
* tool discovery,
* failure recovery,
* or continuous entropy control. 

---

### A Proprietary Vendor Term

Harness Engineering is not unique to any single company.

Both **Anthropic** and **OpenAI** use the term publicly.

Academic preprints are likewise beginning to treat it as a general, cross-platform concept. 

---

### Something That Disappears as Models Improve

Quite the opposite.

Anthropic explicitly argues that as models become more capable, the value of a harness is **redistributed rather than eliminated**.

Some forms of validation become unnecessary.

But planning, verification, handoffs, and state governance become increasingly important as models are entrusted with longer, more expensive, and higher-risk tasks. 

In fact, the space of **interesting harness combinations** is unlikely to shrink as model capabilities advance.

Instead, it will **shift**.

An evaluator that is essential today may become redundant for the next generation of models.

Conversely, new capability frontiers will create entirely new harness requirements. 

---

# Frequently Overlooked Questions

## How Do We Test the Harness Itself?

When we say that a harness makes an agent verifiable, an obvious meta-question arises:

> **How do we verify the harness?**

If an evaluator is itself another large language model—and that evaluator is also prone to hallucination—we have merely constructed a loop in which one unreliable system validates another. 

Anthropic's practical approach is to rely on **computational sensors** whenever possible:

* automated test suites,
* linters,
* static type checkers.

Only subjective judgments—such as visual aesthetics or coding style—are delegated to **inferential sensors**.

This layered strategy is pragmatic, though it is not a perfect solution. 

---

## Emergent Behavior in Multi-Agent Systems

When ten independent agents operate concurrently, the overall system begins to exhibit behaviors that cannot be predicted by analyzing any individual agent in isolation.

This resembles concurrency bugs in distributed systems—but is arguably even more challenging because each "process" is itself nondeterministic.

Current harness designs are still largely optimized for **single-agent** workflows.

Engineering principles for multi-agent harnesses have yet to fully mature. 

---

## Engineering Trade-offs Between Cost and Latency

Every additional layer of a harness—

* planners,
* evaluators,
* sensors,
* garbage collection—

consumes additional tokens and increases latency.

When the overhead introduced by the harness exceeds the quality improvements it provides, the system becomes over-engineered.

How to measure the **return on investment (ROI)** of a harness, and how to dynamically adjust its complexity based on task difficulty, remains an open engineering problem. 

---

# A New Dimension of Security: Attacking Agency Rather Than Data

This is perhaps the most important—and most easily underestimated—issue.

As agents acquire:

* persistent state,
* access to external tools,
* and long-term autonomy,

the attack surface fundamentally changes.

The question is no longer merely:

> **Can the model produce an incorrect answer?**

It becomes:

> **Can the system itself be manipulated into acting on an attacker's behalf?** 

In April 2025, **Invariant Labs** disclosed **Tool Poisoning Attacks**.

Malicious instructions could be hidden inside MCP tool descriptions.

These instructions remain invisible to users while being fully visible to language models, allowing attackers to induce unauthorized behavior.

Only a week later, the researchers demonstrated data exfiltration by chaining an untrusted MCP server with a trusted WhatsApp MCP server.

These findings show that Harness Engineering cannot concern itself solely with throughput and reliability.

It must also address:

* tool trust,
* cross-tool data flow,
* least-privilege access,
* approval boundaries,
* execution isolation. 

Although MCP's open standardization is an important achievement—and Anthropic donated MCP to the Linux Foundation's Agentic AI Foundation in December 2025—the more successful open connectivity becomes, the stronger the governance responsibilities imposed on the harness.

Permission models must evolve beyond simple binary rules ("allowed" vs. "not allowed") toward dynamic policies such as:

* under what conditions an action is permitted,
* to what extent it may proceed,
* and when explicit human approval is required.

In this sense, a harness is not merely the outer loop that improves agent performance.

It is also the **new security boundary**. 

# Conclusion

Looking back over the past four years of AI engineering, we can observe a remarkably consistent pattern.

Each major breakthrough in model capability has been followed by the emergence of a corresponding engineering discipline. 

| Model Capability      | Engineering Discipline |
| --------------------- | ---------------------- |
| Generation            | Prompt Engineering     |
| Connection            | Context Engineering    |
| Reasoning             | Agent Engineering      |
| Long-Running Autonomy | Harness Engineering    |

The deeper pattern is even more revealing.

Each successive discipline shifts the engineering focus **one level outward**:

* from **the prompt**,
* to **the context**,
* to **the execution loop**,
* and ultimately toward **the operating system itself**. 

Harness Engineering should therefore **not** be understood as the final destination.

Rather, it represents a transitional stage.

Its role is to bridge today's application-level agents and tomorrow's **AgentOS**.

Viewed from this perspective, the true significance of Harness Engineering is not that it improves prompting.

Nor is it simply that it makes AI agents more reliable.

Its deeper contribution is that it introduces **governance** into autonomous systems.

As long as models remain incapable of reliably governing themselves, some form of external harness will remain indispensable.

Whether future systems ultimately evolve into fully fledged AgentOS platforms remains uncertain.

Yet one conclusion already appears difficult to dispute:

> **The future bottleneck of AI is no longer intelligence.**
>
> **It is governance.**
